## `POST /v1/account/setup`

Idempotent recovery endpoint that ensures the caller's org has a Stripe customer + envelope + audit row provisioned. The Clerk webhook normally provisions on org creation; this endpoint is the **manual fallback** when the webhook missed delivery (network blip, Svix queue stuck, brand-new account that hasn't had a webhook fire yet).

The same `provisionOrg` code path runs for both flows, so a delayed webhook + a recovery call collapse to **one** envelope, **one** Stripe customer, and **one** audit row.

<Note>
  This endpoint uses Clerk session-token auth, **not** an API key. The data API endpoints (`/v1/address/*`) continue to use `X-Api-Key`.
</Note>

## Authentication

Send the user's Clerk session token as a `Bearer` token in the `Authorization` header. From a Clerk-React frontend you'd do:

```ts
import { useAuth } from "@clerk/clerk-react";

const { getToken } = useAuth();
const token = await getToken();
const res = await fetch("https://api.prontiq.dev/v1/account/setup", {
  method: "POST",
  headers: { Authorization: `Bearer ${token}` },
});
```

### Required JWT claims

| Claim | Where it comes from | Why we need it |
|-------|---------------------|----------------|
| `sub` | Clerk session, automatic | Identifies the user; passed through to `clerkClient.users.getUser(sub)` for primary email lookup |
| `org_id` | **JWT template — see below** | Identifies the org being provisioned |
| `org_role` | **JWT template — see below** | Authorisation gate. Only `org:admin` (and operator-configurable additional roles via `CLERK_ADMIN_ROLES`) can call this endpoint — same role policy the Clerk webhook uses |

<Warning>
  **Operator pre-requisite, BOTH dev and prod tenants:** the Clerk session-token JWT template must include BOTH `org_id` AND `org_role`:

  ```json
  {
    "org_id": "{{org.id}}",
    "org_role": "{{org.role}}"
  }
  ```

  Configure at: Clerk Dashboard → **Sessions** → **Customize session token** → add both fields to the template.

  Without `org_id` the endpoint returns `400 NO_ACTIVE_ORG`. Without `org_role` it returns `400 NO_ROLE_CLAIM`. Both are operator-side fixes — the JWT itself is valid, but the template doesn't expose the claim the gate needs.
</Warning>

<Warning>
  **Frontend pre-requisite:** the user must have called `setActive({ organization })` before invoking this endpoint. Even with the JWT template above, `org_id` and `org_role` are only populated when the session has an active organization.
</Warning>

### Authorization

This endpoint is **org-admin-only**. The same role policy the Clerk webhook uses gates this endpoint:

- Default admin roles: `org:admin` (Clerk's canonical creator role) and `admin` (legacy / custom-role-set fallback).
- Operators can override via the `CLERK_ADMIN_ROLES` environment variable on the `PqAccount` Lambda (comma-separated; same pattern as the webhook). Example: `CLERK_ADMIN_ROLES=owner,principal`.
- Non-admin members (e.g., `org:member` invitees) receive `403 INSUFFICIENT_ROLE`.

This prevents an invited member from racing a delayed/missed webhook and becoming the recorded `ownerEmail`, Stripe customer email, or welcome-email recipient for the org.

## Request

No request body. The user identity + org are read from the verified JWT.

```bash
curl -X POST "https://api.prontiq.dev/v1/account/setup" \
  -H "Authorization: Bearer eyJhbGciOiJSUzI1NiIs..."
```

## Responses

### 201 — Org freshly provisioned

```json
{
  "status": "created",
  "stripeCustomerId": "cus_ABC123...",
  "emailSent": true
}
```

A new Stripe customer was created, the envelope was written to DynamoDB, the audit row was committed in the same transaction, and a welcome email was best-effort sent. `emailSent: false` here means the envelope is durable but the welcome email was skipped or failed non-fatally, for example because the address is currently suppressed after an SES bounce / complaint event or SES temporarily rejected the send.

### 200 — Idempotent replay

```json
{
  "status": "already_exists",
  "stripeCustomerId": "cus_ABC123..."
}
```

The org's envelope already exists. Zero new side-effects: no second Stripe customer, no second envelope, no second audit row. Safe to call from the frontend on every dashboard load.

### 400 — `NO_ACTIVE_ORG`

```json
{
  "error": {
    "code": "NO_ACTIVE_ORG",
    "message": "JWT does not include an org_id claim. Ensure the Clerk session token JWT template includes { \"org_id\": \"{{org.id}}\" } and the frontend has called setActive({ organization }) before invoking this endpoint.",
    "status": 400,
    "request_id": "req_..."
  }
}
```

The JWT verified successfully, but `org_id` is missing or empty. See the two operator pre-requisites above.

### 400 — `NO_ROLE_CLAIM`

```json
{
  "error": {
    "code": "NO_ROLE_CLAIM",
    "message": "JWT does not include an org_role claim. Ensure the Clerk session token JWT template includes { \"org_role\": \"{{org.role}}\" }.",
    "status": 400,
    "request_id": "req_..."
  }
}
```

The JWT verified successfully and has `org_id`, but `org_role` is missing. Operator-side fix: add `{ "org_role": "{{org.role}}" }` to the JWT template (see the JWT template warning above).

### 403 — `INSUFFICIENT_ROLE`

```json
{
  "error": {
    "code": "INSUFFICIENT_ROLE",
    "message": "This action requires an org admin role. Ask your org admin to perform this action.",
    "status": 403,
    "request_id": "req_...",
    "details": { "role": "org:member" }
  }
}
```

The caller is a verified org member but not an admin. Mirrors the Clerk webhook's role gate (which silently skips non-admin `organizationMembership.created` events). User-facing fix: ask your org admin to sign in and complete setup. Operator can override the admin-role set via `CLERK_ADMIN_ROLES` on the `PqAccount` Lambda if your org uses custom roles.

### 401 — `INVALID_TOKEN`

```json
{
  "error": {
    "code": "INVALID_TOKEN",
    "message": "Invalid or expired token",
    "status": 401,
    "request_id": "req_...",
    "details": { "reason": "token-expired" }
  }
}
```

**Caller fault.** Returned for: missing `Authorization` header, missing `Bearer` prefix, expired token, tampered signature, unknown issuer, signature algorithm mismatch.

Distinct from `503 VERIFIER_UNAVAILABLE` below: 401 means **the user's token is bad** (sign in again), while 503 means **Clerk is having trouble** (retry the same request).

### 503 — `VERIFIER_UNAVAILABLE`

```json
{
  "error": {
    "code": "VERIFIER_UNAVAILABLE",
    "message": "Authentication service is temporarily unavailable. Please retry.",
    "status": 503,
    "request_id": "req_...",
    "details": {
      "reason": "clerk_api_503",
      "retryAfter": 30
    }
  }
}
```

**Server-side / upstream outage.** Returned when `verifyToken` couldn't reach Clerk to validate the JWT (network error, Clerk 5xx, rate limit, JWKS endpoint unreachable).

The user's token is fine — **safe to retry the same request**. The dashboard should:
- Honour `details.retryAfter` (seconds) when present (Clerk-supplied rate-limit hint).
- Otherwise back off with jittered exponential delay (start at 1s, max 30s).
- Show the user "we're having trouble — try again" UX, NOT a re-auth prompt.

This signal also fires the operator's `PqAccountErrors` CloudWatch alarm so a sustained Clerk outage is visible without operator polling.

### 500 — `FATAL_FAILURE` (`primary_email_unverified`)

```json
{
  "error": {
    "code": "FATAL_FAILURE",
    "message": "Your primary email address is not verified...",
    "status": 500,
    "request_id": "req_...",
    "details": {
      "reason": "primary_email_unverified",
      "verificationStatus": "unverified"
    }
  }
}
```

The primary email on the user's Clerk profile is not verified. Forwarding an unverified email to Stripe + SES would create a customer record against a typoed/unconfirmed address. **User-facing fix:** verify the email (or set a verified email as primary) in the Clerk profile, then retry.

### 500 — `FATAL_FAILURE` (`user_has_no_primary_email`)

```json
{
  "error": {
    "code": "FATAL_FAILURE",
    "message": "Your account does not have a primary email address...",
    "status": 500,
    "request_id": "req_...",
    "details": { "reason": "user_has_no_primary_email" }
  }
}
```

Phone-first or OAuth-only signup with no email attached. **User-facing fix:** add a primary email in the Clerk profile, then retry.

### 503 — `RETRYABLE_FAILURE`

```json
{
  "error": {
    "code": "RETRYABLE_FAILURE",
    "message": "Provisioning hit a transient error — please retry",
    "status": 503,
    "request_id": "req_..."
  }
}
```

DynamoDB throttle, Stripe 5xx, Clerk Backend API blip, or other transient. **The endpoint is idempotent — safe to retry immediately.** The deterministic Stripe `Idempotency-Key` and the envelope's `attribute_not_exists(apiKeyHash)` condition guarantee that a retry after a partial failure converges on the same single envelope.

## Idempotency

Every call deterministically resolves to the same `ORG#{orgId}` envelope. Concretely:

- **First call** writes the envelope (status 201) and creates a Stripe customer using `Idempotency-Key: org_provision_{orgId}` so a Stripe-side retry doesn't duplicate.
- **Subsequent calls** read the existing envelope (status 200, `already_exists`) and skip the Stripe call entirely.
- **Race with the webhook** (webhook arrives mid-request) is handled by the `attribute_not_exists` condition in the envelope's `TransactWriteItems`. The losing path detects the conditional-check failure, re-reads the envelope, and returns 200 `already_exists`.
