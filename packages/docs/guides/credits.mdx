Prontiq usage is measured in **credits**.

Customers use an API family like **Address API**, and each endpoint call consumes a fixed number of credits. Credits are the abstraction customers see across quotas, usage, and plans.

<Note>
  Credits remain the public usage unit while Prontiq migrates its commercial architecture toward
  Lago. Request-time enforcement counters still live in Prontiq, while Lago is the target billing
  truth.
</Note>

## How credits work

- Credits are the customer-facing usage unit
- Usage is tracked per API key, per API family, per calendar month
- Different endpoints can consume different credits per call
- Monthly quotas and usage all roll up through credits

<Note>
  The live platform still includes a legacy paid-tier ladder during migration. Free-tier examples
  below are current and accurate; paid-plan packaging is still transitioning toward the target Free
  + PAYG model.
</Note>

## Address API credit scoring

Current Address endpoint costs:

| Endpoint        | Credits / Call |
| --------------- | -------------- |
| Autocomplete    | 1              |
| Validate        | 1              |
| Postcode Lookup | 1              |
| Suburb Lookup   | 1              |
| Reverse Geocode | 2              |
| Enrich          | 3              |

## Why scoring differs by endpoint

Different endpoints do different amounts of work, so they consume different credits.

- **Autocomplete** is optimized for high-frequency UI usage
- **Validate** resolves one best match from a full address string
- **Lookups** return compact aggregate answers
- **Reverse** does proximity-based address matching
- **Enrich** returns the fullest Address response shape

## Monthly quota examples

If you are on Free with **10,000 credits/month**:

- 10,000 autocomplete calls = 10,000 credits
- 5,000 reverse calls = 10,000 credits
- 3,333 enrich calls = 9,999 credits

Legacy paid tiers in the live runtime do not hard-fail at the quota boundary in
the same way as Free. They can continue into overage on the current migration-era
commercial path.

## Where credits show up

- Plan docs describe included credits
- Endpoint docs link back to this page for the scoring model
- Response headers show remaining quota in credits
- Usage is tracked in credits, not raw HTTP request count
- Prontiq can send quota emails at 80% and 100% usage for the active month
- Lago billing events receive the same credit delta asynchronously after
  request-time enforcement succeeds

## Quota emails

Prontiq can notify account owners when usage crosses key monthly thresholds:

- **80% of included credits**: warning email
- **100% of included credits**: limit / overage email

These emails are sent at the API-family level and are based on consumed credits, not raw request count.

<Info>
  Credits should be transparent and predictable. If one endpoint consumes more credits than another,
  that should be visible here and in the associated product documentation.
</Info>
