## Monthly quotas

Prontiq tracks monthly quota in credits. Usage is tracked per API key, per API family, and different endpoints can consume different credit amounts.

### Live today

| Plan Surface      | Credits / Month | Runtime behavior at quota boundary |
| ----------------- | --------------- | ---------------------------------- |
| Free              | 10,000          | Hard `429 QUOTA_EXCEEDED`          |
| Legacy paid tiers | tier-dependent  | Request proceeds as overage        |

The current live paid path still includes legacy tiers in runtime (`payg`,
`starter`, `growth`, `max`, `enterprise`). Those tiers do **not** hard-fail at
the monthly quota boundary; they continue and can accrue overage on the current
migration-era commercial path.

### Target direction

| Plan Surface | Credits / Month                                 |
| ------------ | ----------------------------------------------- |
| Free         | 10,000                                          |
| PAYG         | No hard monthly cap; billed by consumed credits |

<Info>
  Pricing and commercial packaging are documented on the Prontiq homepage and the Billing guide.
  Mintlify docs focus on how quota and credit consumption behave at the API layer.
</Info>

## How it works

- Usage is counted per API key, per API family, per calendar month
- Endpoint scoring is expressed in credits, for example Address Enrich consumes more credits than Address Autocomplete
- On the Free plan, exceeding quota returns `429 QUOTA_EXCEEDED`
- On the current legacy paid path, requests can continue into overage instead of hard-failing with `429`
- The response includes `resets_at` with the next month's start date
- Quotas reset at midnight UTC on the 1st of each month
- Prontiq can send quota emails at 80% and 100% of monthly included credits

## API Gateway throttling

In addition to monthly quotas, API Gateway enforces per-route rate limits to protect the infrastructure:

| Limit     | Value               |
| --------- | ------------------- |
| Burst     | 100 requests/second |
| Sustained | 50 requests/second  |

These limits apply across all API keys. If you need higher throughput, contact us.

## Per-key burst limiting

Prontiq also enforces a per-key burst limiter in the application layer.

- It uses a token bucket keyed by your API key hash
- Bucket capacity is based on your plan's per-second rate limit
- If the bucket is empty, the API returns `429 RATE_LIMITED`
- Responses include a `Retry-After` header so clients know when to retry

Current live per-key burst defaults:

| Plan Surface      | Requests / Second |
| ----------------- | ----------------- |
| Free              | 10                |
| Legacy PAYG       | 25                |
| Legacy Starter    | 50                |
| Legacy Growth     | 100               |
| Legacy Max        | 250               |
| Legacy Enterprise | Custom / uncapped |

<Info>
  The burst limiter is enforced per Lambda instance, not globally across every concurrent instance.
  API Gateway throttling still provides the outer global safety net.
</Info>

<Info>
  Responses already include `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`, and
  `X-RateLimit-Product` so customers can track remaining monthly credits in real time.
</Info>
