Prontiq uses a **Lago-centered** commercial architecture.

The target model is:

- **Lago** owns plans, pricing, invoicing, included units, PAYG/package
  behavior, and entitlement truth
- **Stripe** handles payment processing only
- **Prontiq** keeps request-time credit enforcement in DynamoDB and emits billing events asynchronously

## What customers see

- Credits remain the public usage unit
- Free and PAYG are the target commercial direction
- API response headers still reflect request-time quota state from Prontiq
- Plan and quota changes are configured in Lago and projected into Prontiq's
  local enforcement state

<Note>
  The live platform still contains a legacy paid-tier ladder during migration. Public docs in this
  section describe the target commercial direction and should be read alongside live-runtime notes
  in the Credits and Rate Limits guides.
</Note>

## What the platform owns

Prontiq still owns:

- API key lifecycle
- org-scoped account identity via Clerk `orgId`
- real-time credit enforcement
- per-key usage tracking
- billing-event emission away from the hot path

## Runtime posture

The live platform no longer owns a direct Stripe billing runtime:

- Lago owns the customer-facing billing state.
- Stripe is configured inside Lago as the payment rail.
- No API usage changes are required for data endpoints.

## Customer identity

### Live contract

Commercial/account identity is represented by:

- the Clerk organization
- the Prontiq org envelope in DynamoDB
- Lago commercial state

The active platform customer contract is Clerk `orgId`. Lago customer
`external_id` is the same `orgId`, and Lago subscriptions use
`lago_sub_${orgId}`. Lago provider-owned `lago_id` values and Stripe `cus_...`
IDs are linkage fields only; they are not Prontiq customer identity.

<Note>
  Legacy `pq_cust_*` records from P1B.14-P1B.21 are retained only as migration evidence.
  Active runtime paths use Clerk `orgId`.
</Note>

## Usage event forwarding

Prontiq forwards billing events to Lago asynchronously. The API request path
does not call Lago. Each accepted event uses a deterministic transaction ID and
sends the credit delta for the request so replay cannot double-bill when Lago is
configured with the matching subscription and metric.

## Console billing surfaces

Signed-in console billing screens must not call Lago or Stripe directly from
the browser. Billing reads and payment-link actions use a Vercel server-side
BFF. Replay-safe plan changes use the private account API with Clerk JWT auth
and a per-request idempotency key.

API-key users should continue to integrate through the public data API and use
the console UI for account billing actions.

Plan changes from the console are Lago-backed. Prontiq records replay-safe
action evidence, asks Lago to change the subscription, and waits for Lago
webhook reconciliation to project the new plan into request-time enforcement.
