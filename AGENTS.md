## Build & Test Commands

```bash
pnpm install          # Install all dependencies (workspace-aware)
pnpm build            # Build all packages (via Turborepo)
pnpm typecheck        # Type-check all packages
pnpm lint             # Lint all packages
pnpm test             # Run all tests
pnpm dev              # Start SST dev mode (live Lambda)

# Per-package
pnpm --filter @prontiq/api build
pnpm --filter @prontiq/shared typecheck
```

The scaffolded frontend apps default `NEXT_PUBLIC_API_URL` to
`https://api.prontiq.dev` for local build/typecheck/dev when the variable is
unset. Override it explicitly only when pointing `apps/landing` or
`apps/console` at a different API host.

## Architecture

**Unified data API platform** for Australian and global open data. See `ARCHITECTURE.MD` for the full design.

**Pattern:** Free open dataset -> independent pipeline -> S3 (NDJSON + manifest.json) -> event-driven indexing -> OpenSearch -> commercial API -> auth / billing / docs / SDKs.

**Stack (live):** SST v4 + Pulumi, Hono + @hono/zod-openapi on Lambda (ARM64), OpenSearch, DynamoDB. SDKs auto-generated by Speakeasy from committed OpenAPI spec. **Two ingress paths to org provisioning** are live in prod, both calling `createProvisioningService().provisionOrg(...)` from `@prontiq/control-plane`: (1) Clerk webhook (`POST /webhooks/clerk`) verifies Svix signature, gates on org admin role; (2) `POST /v1/account/setup` (Clerk-JWT-authenticated) is the dashboard recovery path when the webhook missed delivery. Same idempotency invariant; any combination of webhook + recovery calls collapses to one Prontiq customer envelope, one Lago Free subscription in forward mode, and one audit row. Legacy Stripe customer creation is rollback-only behind `LEGACY_STRIPE_RUNTIME_ENABLED=true`. API keys are DynamoDB-native (SHA-256 hash-based; see ADR-001, ARCHITECTURE.MD Â§5.5).

**Commercial architecture posture:** P1B.19 cuts runtime billing over to the Lago-centered model: Lago is the commercial system of record, Stripe is the payment rail only, Prontiq keeps hot-path credit enforcement in DynamoDB, and SQS buffers billing events away from the request path. `customerId` is the platform-owned customer identity (`pq_cust_<ulid>`); Clerk org IDs, Stripe customer IDs, and Lago `lago_id` values are linkage fields only. The P1B.15 SQS producer is feature-flagged by `BILLING_EVENTS_ENABLED`; the P1B.16 worker forwards queued events to Lago with `transaction_id = eventId`, `external_subscription_id = pq_sub_<customer ulid>`, and `properties.credits = creditDelta`. P1B.17 adds `POST /webhooks/lago` for async subscription/invoice reconciliation back into local enforcement state, keyed by `X-Lago-Unique-Key` and gated by `LAGO_WEBHOOK_RECONCILIATION_ENABLED`. P1B.18a certified live Lago smoke in dev/prod on 2026-04-26; use `pnpm --filter @prontiq/control-plane lago:smoke:event` or an equivalent production-contract helper for future smoke billing events, and do not hand-build Lago transaction IDs. P1B.19 sets `LEGACY_STRIPE_RUNTIME_ENABLED=false` for cutover: Stripe webhooks verify signatures but return `retired`, and the legacy Stripe cron/month-close return disabled summaries. Retained prod smoke fixtures should remain available through P1B.20 but must be labelled/inventoried as test-only; final destructive cleanup is P1B.21 after P1B.20. Keep Lago and Stripe off the API hot path.

**P1B.19 status:** implemented for review. Dev/prod webhook smoke completed with unique keys `prontiq-platform-dev-smoke-20260426T051602Z` and `prontiq-platform-prod-smoke-20260426T051812Z`; replay returned `200 duplicate`. P1B.19 changes the post-cutover deployed posture to `COUNTER_PERIOD_SOURCE=lago` after verification, with calendar fallback retained for rollback.

**Observability (current state):** CloudWatch alarms/dashboard + SNS email remain the AWS-native operations plane. Honeycomb backend tracing is now live and verified for deployed Lambdas behind `HONEYCOMB_API_KEY`, and `PqApi` X-Ray remains in place as a retained secondary trace path. Browser/frontend telemetry is not part of this ticket.

**SES deliverability (current state):** suppression handling and simulator-verified SES flows are live, but normal-recipient production readiness is owned by `P1B.08a`. Do not treat DKIM alone as complete; custom MAIL FROM on `bounce.prontiq.dev`, DMARC relaxed SPF alignment, SES production-access approval, and a normal-recipient test send are required.

**Frontend (current state):** `apps/landing` and `apps/console` now have Tailwind v3.4, app-local shadcn/ui primitives, dark mode, responsive shell foundations, and frontend Vitest + Testing Library. `apps/landing` is now a real conversion surface: proxy-backed live autocomplete demo, config-owned free tier, and Clerk modal CTA. Forward-looking commercial UX should assume Lago-backed billing surfaces in `apps/landing` and `apps/console`, with any remaining Stripe-hosted behavior treated as legacy migration context. `apps/console` carries a real Clerk auth boundary that is enabled only when the Clerk env keys are present; keyless fallback is allowed only through the local/CI frontend helper path, and missing Clerk keys otherwise fail closed.

## Constraints

- **ESM only.** All packages use `"type": "module"`. Import extensions must use `.js`.
- **Strict TypeScript.** `noUncheckedIndexedAccess`, `noUnusedLocals`, `noUnusedParameters` enabled.
- **No vendor in the hot path.** API key verification is a single DynamoDB `GetItem` by SHA-256 hash. See `ARCHITECTURE.MD` Â§5.5 and ADR-001.
- **Customer identity is platform-owned.** Use `customerId` for commercial customer identity. Do not make Clerk `orgId`, Stripe `cus_...`, or Lago `lago_id` the source of truth.
- **Billing events are queue-first.** API handlers must never call Lago directly. Emit `BillingUsageEventV1` only after DynamoDB enforcement succeeds, and preserve deterministic `eventId` inputs for idempotency.
- **Legacy Stripe runtime is gated.** `LEGACY_STRIPE_RUNTIME_ENABLED=false` is the cutover posture. Do not add new calls to Stripe provisioning, Stripe billing cron, or Stripe month-close outside explicit rollback handling.
- **Lago forwarding is minimal.** The worker may send only Lago-required usage fields: `transaction_id`, `external_subscription_id`, metric `code`, timestamp, and credit properties. Do not forward API-key hashes, key prefixes, request URLs, headers, IPs, user agents, query strings, or response payloads.
- **OpenAPI spec is the single source of truth.** Routes are defined with `@hono/zod-openapi` â€” never add API endpoints without Zod schemas.
- **Public/private API specs stay separate.** `packages/docs/openapi.json` is the public Mintlify/Speakeasy data API spec. Clerk-authenticated `/v1/account/*` console/admin routes belong only in `packages/api/openapi.private.json` via `packages/api/src/openapi-private.ts`.
- **Manifest contract is sacred.** See `ARCHITECTURE.MD` section 5.1.2. Pipelines produce NDJSON + manifest. Platform consumes.
- **Index names follow `{product}-{version}` convention.** Aliases are from the product registry. Never hardcode alias names in route handlers.

## Testing Patterns

- **Framework:** `node:test` is the default for backend, infrastructure, and utility packages. Frontend apps (`apps/landing`, `apps/console`) now use Vitest + Testing Library for unit/component coverage. Existing backend package tests run after a per-package build via `pnpm -w build --filter @prontiq/<pkg>... && node --test dist/<file>.test.js`. See any package's `package.json` `test` script for the canonical invocation.
- **Test files:** Co-located as `*.test.ts` next to source files. Integration tests use the `*.integration.test.ts` suffix and run separately via the `test:integration` script.
- **Mocking:** Dependency injection at the service boundary. `createProvisioningService({ ddb, stripe, sleep, logger, sendWelcomeEmail })`, `createClerkHandler({ service, webhookSecret, clerkClient, adminRoles })` â€” every dependency is overridable for tests. Never mock SDK clients directly.
- **Integration tests** hit real DynamoDB Local (`amazon/dynamodb-local:2.5.2`) and real OpenSearch via the `integration-test` job's service containers in CI. Locally: `docker run -p 8000:8000 amazon/dynamodb-local:2.5.2` then `pnpm --filter @prontiq/<pkg> test:integration`.
