# AGENTS.md

## Build & Test Commands

```bash
pnpm install          # Install all dependencies (workspace-aware)
pnpm build            # Build all packages (via Turborepo)
pnpm typecheck        # Type-check all packages
pnpm lint             # Lint all packages
pnpm test             # Run all tests
pnpm dev              # Start SST dev mode (live Lambda)

# Per-package
pnpm --filter @prontiq/api build
pnpm --filter @prontiq/shared typecheck
```

## Architecture

**Unified data API platform** for Australian and global open data. See `ARCHITECTURE.MD` for the full design.

**Pattern:** Free open dataset -> independent pipeline -> S3 (NDJSON + manifest.json) -> event-driven indexing -> OpenSearch -> commercial API -> auth / billing / docs / SDKs.

**Stack (live):** SST v4 + Pulumi, Hono + @hono/zod-openapi on Lambda (ARM64), OpenSearch, DynamoDB. SDKs auto-generated by Speakeasy from committed OpenAPI spec. **Two ingress paths to org provisioning** are live in prod, both calling `createProvisioningService().provisionOrg(...)` from `@prontiq/control-plane`: (1) Clerk webhook (`POST /webhooks/clerk`) verifies Svix signature, gates on org admin role; (2) `POST /v1/account/setup` (Clerk-JWT-authenticated) is the dashboard recovery path when the webhook missed delivery. Same idempotency invariant; any combination of webhook + recovery calls collapses to one envelope, one Stripe customer, one audit row. API keys are DynamoDB-native (SHA-256 hash-based; see ADR-001, ARCHITECTURE.MD §5.5).

**Stack (planned — remaining P1B / all P1C):** Stripe subscription webhook (P1B.06), SES bounce handler (P1B.08), billing cron (P1B.10), month-close (P1B.11), `/account` dashboard (P1C). The `packages/webhooks/src/stripe.ts` file is still a stub awaiting P1B.06.

## Monorepo Structure

```
packages/
  shared/         -> @prontiq/shared          Types, constants, Zod schemas (dep-light; depended on by everything)
  control-plane/  -> @prontiq/control-plane   provisionOrg service + writeAudit helpers + resolvePrimaryEmail (consumed by webhooks AND api)
  api/            -> @prontiq/api             Hono API on Lambda (address routes via index.ts $default Lambda + /v1/account/setup via separate account-handler.ts PqAccount Lambda)
  ingestion/      -> @prontiq/ingestion       Step Functions + Lambda bulk indexing
  webhooks/       -> @prontiq/webhooks        Clerk webhook handler (live in prod) + Stripe stub (P1B.06 pending)
  docs/           -> @prontiq/docs            Mintlify documentation
  plugins/
    shopify/      -> Checkout UI Extension
    woocommerce/  -> WP plugin
    web-component/ -> <prontiq-address> widget
```

**Dependency graph:**
- `shared` -> `control-plane`, `api`, `ingestion`, `webhooks`, `plugins/*`
- `control-plane` -> `webhooks`, `api`

## Constraints

- **ESM only.** All packages use `"type": "module"`. Import extensions must use `.js`.
- **Strict TypeScript.** `noUncheckedIndexedAccess`, `noUnusedLocals`, `noUnusedParameters` enabled.
- **No vendor in the hot path.** API key verification is a single DynamoDB `GetItem` by SHA-256 hash. See `ARCHITECTURE.MD` §5.5 and ADR-001.
- **OpenAPI spec is the single source of truth.** Routes are defined with `@hono/zod-openapi` — never add API endpoints without Zod schemas.
- **Manifest contract is sacred.** See `ARCHITECTURE.MD` section 5.1.2. Pipelines produce NDJSON + manifest. Platform consumes.
- **Index names follow `{product}-{version}` convention.** Aliases are from the product registry. Never hardcode alias names in route handlers.

## Testing Patterns

- **Framework:** `node:test` (Node's built-in runner; no Vitest in this repo). Tests run after a per-package build via `pnpm -w build --filter @prontiq/<pkg>... && node --test dist/<file>.test.js`. See any package's `package.json` `test` script for the canonical invocation.
- **Test files:** Co-located as `*.test.ts` next to source files. Integration tests use the `*.integration.test.ts` suffix and run separately via the `test:integration` script.
- **Mocking:** Dependency injection at the service boundary. `createProvisioningService({ ddb, stripe, sleep, logger, sendWelcomeEmail })`, `createClerkHandler({ service, webhookSecret, clerkClient, adminRoles })` — every dependency is overridable for tests. Never mock SDK clients directly.
- **Integration tests** hit real DynamoDB Local (`amazon/dynamodb-local:2.5.2`) and real OpenSearch via the `integration-test` job's service containers in CI. Locally: `docker run -p 8000:8000 amazon/dynamodb-local:2.5.2` then `pnpm --filter @prontiq/<pkg> test:integration`.

## Do NOT

- Don't use `any` type — use `unknown` with type narrowing
- Don't import without `.js` extension (ESM requirement)
- Don't add `console.log` — use structured logging
- Don't put secrets in code — secrets flow GitHub Environment secrets → workflow `env:` block → `process.env.X` → `sst.config.ts` reads → wrap with `$util.secret()` before passing to Lambda environment so Pulumi state stores them encrypted. See `docs/runbooks/clerk-webhook.md` § preconditions and the existing `WELCOME_EMAIL_FROM` / `STRIPE_SECRET_KEY` patterns. Do NOT use `sst.Secret()` — it conflicts with the GitHub-Environment pattern (see ADR-002 § hardening contract #4 for rationale)
- Don't query OpenSearch by versioned index name — always use the alias
- Don't add dependencies to the root package.json unless they're truly shared dev tools
- Don't modify the manifest contract without updating `ARCHITECTURE.MD` section 5.1.2
