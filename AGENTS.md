## Architecture

**Unified data API platform** for Australian and global open data. See `ARCHITECTURE.MD` for the full design.

**Pattern:** Free open dataset -> independent pipeline -> S3 (NDJSON + manifest.json) -> event-driven indexing -> OpenSearch -> commercial API -> auth / billing / docs / SDKs.

**Stack (live):** SST v4 + Pulumi, Hono + @hono/zod-openapi on Lambda (ARM64), OpenSearch, DynamoDB. SDKs auto-generated by Speakeasy from committed OpenAPI spec. **Two ingress paths to org provisioning** are live in prod, both calling `createProvisioningService().provisionOrg(...)` from `@prontiq/control-plane`: (1) Clerk webhook (`POST /webhooks/clerk`) verifies Svix signature, gates on org admin role; (2) `POST /v1/account/setup` (Clerk-JWT-authenticated) is the dashboard recovery path when the webhook missed delivery. Same idempotency invariant; any combination of webhook + recovery calls collapses to one Clerk-org-owned Prontiq org envelope, one Lago Free subscription, and one audit row. API keys are DynamoDB-native (SHA-256 hash-based; see ADR-001, ARCHITECTURE.MD Â§5.5).

**Commercial architecture posture:** P1B.22 pivots the active commercial identity to the Clerk organization id. Lago is the commercial system of record, Stripe is the payment rail only through Lago, Prontiq keeps hot-path credit enforcement in DynamoDB, and SQS buffers billing events away from the request path. Lago customer `external_id = orgId`; Lago subscription `external_id = lago_sub_${orgId}`. P1B.24 makes Lago effective charges and entitlements the source for plan limits; local DynamoDB fields are only an enforcement projection. Lago plan codes are dynamic strings and must not require TypeScript tier edits. The API producer emits `BillingUsageEventV2` with `orgId` after local DynamoDB enforcement succeeds. Active AWS private account routes are setup recovery, key management, usage, and replay-safe plan changes: `POST /v1/account/setup`, `GET /v1/account/status`, `GET /v1/account/keys`, `POST /v1/account/keys/create`, `POST /v1/account/keys/rotate`, `POST /v1/account/keys/revoke`, `GET /v1/account/usage`, and `POST /v1/account/billing/plan-change`. Other `/v1/account/billing*` routes are retired. `customerId`, `pq_cust_*`, `pq_sub_*`, and `prontiq-customers` are legacy P1B.14-P1B.21 evidence only. Keep Lago and Stripe off the address API hot path.

**P1B.19/P1B.20/P1B.21 status:** P1B.19 cutover is complete. P1B.20 removed the legacy Stripe webhook, billing cron, month-close, pricing table, Stripe envs, and Stripe package dependencies from active deploys. P1B.21 left `BILLING_EVENTS_ENABLED=true`, `COUNTER_PERIOD_SOURCE=lago`, and `LAGO_WEBHOOK_RECONCILIATION_ENABLED=true` for prod go-live.

**Post-deploy smoke posture:** P1F.04 wires the public Address API smoke into dev/prod deploy workflows. The smoke uses a stage-owned labelled API key stored only as the GitHub Environment secret `PRONTIQ_KEY`; never print raw keys or hashes in logs/docs. Prod private account/Clerk smokes remain manual because prod Clerk session minting is not durable. Do not reuse retired prod smoke key prefixes `pq_live_4a85`, `pq_live_03f7`, or `pq_live_0300`.

**Observability (current state):** CloudWatch alarms/dashboard + SNS email remain the AWS-native operations plane. Honeycomb backend tracing is now live and verified for deployed Lambdas behind `HONEYCOMB_API_KEY`, and `PqApi` X-Ray remains in place as a retained secondary trace path. Browser/frontend telemetry is not part of this ticket.

**SES deliverability (current state):** suppression handling and simulator-verified SES flows are live. Custom MAIL FROM on `bounce.prontiq.dev` and DMARC relaxed SPF alignment are configured. Normal-recipient production readiness remains owned by `P1B.08a`; SES production-access approval and a normal-recipient test send are still required.

**Frontend (current state):** `apps/landing` and `apps/console` now have Tailwind v3.4, app-local shadcn/ui primitives, dark mode, responsive shell foundations, and frontend Vitest + Testing Library. `apps/landing` is now a real conversion surface: proxy-backed live autocomplete demo, config-owned Free/PAYG pricing cards, and Clerk modal CTA. The Stripe Pricing Table path has been removed. Forward-looking commercial UX should assume Lago-backed billing surfaces in `apps/landing` and `apps/console`. `apps/console` carries a real Clerk auth boundary that is enabled only when the Clerk env keys are present; keyless fallback is allowed only through the local/CI frontend helper path, and missing Clerk keys otherwise fail closed.

**Stack (planned â€” remaining P1C and later):** real landing and console product surfaces on top of the live frontend base.

## Constraints

- **ESM only.** All packages use `"type": "module"`. Import extensions must use `.js`.
- **Strict TypeScript.** `noUncheckedIndexedAccess`, `noUnusedLocals`, `noUnusedParameters` enabled.
- **No vendor in the hot path.** API key verification is a single DynamoDB `GetItem` by SHA-256 hash. See `ARCHITECTURE.MD` Â§5.5 and ADR-001.
- **Customer identity is Clerk-org-owned.** Use Clerk `orgId` as the active commercial customer identity. Do not reintroduce generated `pq_cust_*`, Stripe `cus_...`, or Lago `lago_id` as the source of truth.
- **Billing events are queue-first.** Address API handlers must never call Lago directly. Emit `BillingUsageEventV2` only after DynamoDB enforcement succeeds, and preserve deterministic `eventId` inputs for idempotency. The private account plan-change route is the explicit low-QPS exception and must use the billing-action ledger before crossing the Lago boundary.
- **Lago owns plan limits.** Runtime enforcement reads denormalized Lago projection fields on org/key records first. Local `PLANS` is only a compatibility fallback for legacy tier rows that pre-date projection fields; unknown dynamic Lago plan codes without a projection fail conservatively to the Free hard-cap posture until reconciliation succeeds.
- **Lago projection must fail safe.** Enabled address plans require a positive-integer `address_api.rate_limit_per_second`; invalid or missing bouncer entitlements are drift and must preserve last-known-good local rows. Reconciliation must refresh the full org envelope projection even when no active keys exist, because future keys inherit from the envelope.
- **Provisioning must not create productless orgs.** For a missing org, `provisionOrg` validates the Lago Free subscription projection before the first `ORG#{orgId}` TransactWrite. If Lago projection fails, return retryable failure without committing the envelope. Existing incomplete envelopes are repaired by replay.
- **No platform-owned Stripe runtime.** Stripe is only the payment rail configured inside Lago. Do not add Stripe provisioning, Stripe webhook, Stripe billing cron, Stripe month-close, or Stripe Pricing Table code back to this repo without a new decision record.
- **Lago forwarding is minimal.** The worker may send only Lago-required usage fields: `transaction_id`, `external_subscription_id`, metric `code`, timestamp, and credit properties. Do not forward API-key hashes, key prefixes, request URLs, headers, IPs, user agents, query strings, or response payloads.
- **OpenAPI spec is the single source of truth.** Routes are defined with `@hono/zod-openapi` â€” never add API endpoints without Zod schemas.
- **Public/private API specs stay separate.** `packages/docs/openapi.json` is the public Mintlify/Speakeasy data API spec. Clerk-authenticated `/v1/account/*` console/admin routes belong only in `packages/api/openapi.private.json` via `packages/api/src/openapi-private.ts`.
- **Account admin gates are per-route.** `clerkJwt()` is Lambda-wide for `/v1/account/*`; mutation routes apply `clerkAdminOnly()` inside their route factory. Member-allowed reads such as `/v1/account/status` and `/v1/account/keys` must not inherit admin-only middleware accidentally.
- **Manifest contract is sacred.** See `ARCHITECTURE.MD` section 5.1.2. Pipelines produce NDJSON + manifest. Platform consumes.
- **Index names follow `{product}-{version}` convention.** Aliases are from the product registry. Never hardcode alias names in route handlers.

## Testing Patterns

- **Framework:** `node:test` is the default for backend, infrastructure, and utility packages. Frontend apps (`apps/landing`, `apps/console`) now use Vitest + Testing Library for unit/component coverage. Existing backend package tests run after a per-package build via `pnpm -w build --filter @prontiq/<pkg>... && node --test dist/<file>.test.js`. See any package's `package.json` `test` script for the canonical invocation.
- **Test files:** Co-located as `*.test.ts` next to source files. Integration tests use the `*.integration.test.ts` suffix and run separately via the `test:integration` script.
- **Mocking:** Dependency injection at the service boundary. `createProvisioningService({ ddb, lagoClient, lagoPaymentProviderCode, sleep, logger, sendWelcomeEmail })`, `createClerkHandler({ service, webhookSecret, clerkClient, adminRoles })` â€” every dependency is overridable for tests. Never mock SDK clients directly.
- **Integration tests** hit real DynamoDB Local (`amazon/dynamodb-local:2.5.2`) and real OpenSearch via the `integration-test` job's service containers in CI. Locally: `docker run -p 8000:8000 amazon/dynamodb-local:2.5.2` then `pnpm --filter @prontiq/<pkg> test:integration`.
